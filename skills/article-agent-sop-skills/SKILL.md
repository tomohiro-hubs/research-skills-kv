# Claude Codeは“プロンプト”じゃない。SkillsでSOP化するとAIはチームになる

このスキルは、「プロンプト」による指示ではなく「SOP（Standard Operating Procedure）」としてのスキル定義を行うことで、AIエージェントの再現性を高めるための記事執筆を支援します。

## 記事の構成要素 (Core Components)

### 1. タイトル / アングル (Title/Angle)
- **Claude Codeは“プロンプト”じゃない。SkillsでSOP化するとAIはチームになる**
- 狙い: 単なる「AIの使い方」を超え、組織論や業務フローの設計としてのAI活用を提案する。

### 2. 主張 (Claim)
- **SkillsをSOP（役割/手順/検査点）として固定すると、AIの再現性が上がり「任せられる仕事」が増える。**
- 理由: プロンプトは都度指示だが、SOPは業務の標準化であり、エージェントを「部下」や「同僚」へと昇華させるため。

### 3. 具体的な論点 (Specifics)
以下の3点を中心に記事を構成します。

#### A. 役割を分ける（設計/実装/レビュー）
- **課題**: 「これ作って」という全部盛りの指示は、AIのコンテキストあふれや暴走を招く。
- **解決策**: 人間のチーム開発と同様に、工程ごとに役割（ロール）を定義する。
- **SOP化**: 「設計担当」→「実装担当」→「レビュー担当」のように、異なる視点でエージェントを切り替える（あるいは思考モードを切り替える）手順を確立する。

#### B. 手順を固定する（Discovery -> Planning -> Building）
- **課題**: いきなりコードを書き始めると、要件の抜け漏れで手戻りが発生する。
- **解決策**: 開発フローを強制的にフェーズ分けする。
- **SOP化**:
    1.  **Discovery**: 既存コードの調査、依存関係の特定。
    2.  **Planning**: 実装方針の策定、影響範囲の特定。
    3.  **Building**: コーディングとテスト。
    - いま何を決める時間かを確定することで、AIの迷走を防ぐ。

#### C. Stop & Checkを入れる
- **課題**: AIは間違った方向にも全力で走ってしまう。
- **解決策**: フェーズの切り替わりで必ず人間が確認するポイント（関所）を設ける。
- **SOP化**: 「決まったこと / 未決事項 / 次にやること」を毎フェーズの最後に出力させ、人間が「Go」を出さないと進めない仕組みにする。

## 執筆手順 (Workflow)

1.  **リサーチ**: 現在の「プロンプトエンジニアリング」の限界や、エージェント型AIの失敗例（暴走、ハルシネーション）を収集する。
2.  **SOPの具体例作成**: 実際にClaude Code等で使える簡単なSOP（Markdown形式の指示書など）の例を作成する。
3.  **記事執筆**: 上記の構成要素に基づき、読者が「明日から自分のプロンプトをSOPに変えてみよう」と思えるように執筆する。
4.  **レビュー**: 「チームになる」というメタファーが機能しているか確認する。
